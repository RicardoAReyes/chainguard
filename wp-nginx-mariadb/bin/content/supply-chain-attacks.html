What the Dataset Proves

1. Most attacks happen before deployment

Source and build-stage compromises account for the majority of major software supply chain incidents in the dataset. This reinforces a simple reality: runtime security alone is too late. Federal organizations need controls that establish trust earlier in the software lifecycle.

2. Different stages require different controls

A single control does not solve software supply chain risk. Each stage requires a different trust mechanism:

  • Source: SBOM transparency, curated packages, rebuilt-from-source artifacts
  • Build: Provenance, signing, and tamper-evident CI/CD pipelines
  • Distribution: Immutable artifacts and policy enforcement
  • Deployment: Minimal images, continuous patching, and runtime controls

3. Federal mandates already point in this direction

This model aligns directly to current Federal cybersecurity priorities:

  • EO 14028 / OMB M-22-18: SBOMs and software integrity evidence
  • Continuous ATO: Signed, attestable artifacts that support ongoing authorization evidence
  • POA&M Reduction: Eliminating vulnerable components upstream instead of managing inherited backlog later

Executive Takeaway

Every major software supply chain attack maps to a broken trust boundary. Chainguard addresses those trust boundaries directly with rebuilt, signed, attestable, and minimal software—helping Federal organizations move from reactive vulnerability management to verifiable software integrity.
